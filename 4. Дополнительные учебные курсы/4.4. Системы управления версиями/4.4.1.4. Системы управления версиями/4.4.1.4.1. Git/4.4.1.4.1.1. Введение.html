История возникновения Git

Git был разработан сообществом программистов, разрабатывающим ядро операционной системы "Linux", на основе другой распределенной системы управления версиями - BitKeeper. BitKeeper был разработан в 2002 году специально для разработки ОС Linux, до этого времени все изменения в коде программы передавались разработчикам в виде патчей (заплаток, изменений) и файлами кода.

Но в 2005 году соглашение о бесплатном использовании BitKeeper разработчиками Linux было разорвано, что побудило сообщество разработчиков, начать создание собственной системы управления версиями и паралельной разработки. В основу разработки Git были заложены следующие требования:

  • быстродействие системы;
  • распределенность и независимость ветвей проекта;
  • нелинейность системы;

Первая версия Git увидела свет 7 апреля 2005 года и с того времени обрела огромную популярность и на данный момент используется для разработки таких крупных проектов как: Android, PHP, Jquery, Drupal и многие другие.

1.1. Принцип работы Git с версиями файлов (красным отмечены измененные файлы)

Git является распределенной системой управления версиями (DVCS). В момент клонирования удаленного репозитория, локальная версия Git получает не только все файлы проекта, но и удаленную базу данных изменений и веток, и поэтому для выполнения большинства операций Git не требует подключения к сети и может выполнять все действия используя только локальные ресурсы. В Git существует два типа репозиториев: локальный, расположенный на текущей рабочей станции, и удаленный, расположенный на другом компьютере или сервере. Git может функционировать и без удаленного репозитория, но локальный репозиторий всегда должен существовать. Практически все операции Git производит именно с локальным репозиторием. С удаленным репозиториев Git в основном взаимодействует при синхронизации репозиториев, в момент отправки данных на удаленный репозиторий (push), получния данных с удаленного репозитория (pull) и клонирования удаленного репозитория (clone). В случае недоступности или выхода из строя одного из репозиториев данные проекта можно клонировать из рабочих репозиториев, достигая тем самым высокого уровня отказоустойчивости и сохранности данных проекта. Почти при всех действиях при работе с Git происходит добавление данных в репозиторий практически невозможно случайно удалить данные из базы. Незафиксированные данные можно потерять, но при фиксации данных в Git удалить запись становится очень трудно. Таким образом Git открывает большие возможности для экспериментво без опасения произвести необратимые изменения в проекте.

Архитектурно Git является программой, управление которой происходит коммандами консоли. Все графические интерфейсы для работы с Git являются графическими оболочками, интерпритирующими действия пользователей в команды консоли.

Описание принципов работы Git

Файлы в проекте, использующем Git в качестве системы контроля версий, могут находиться в четырех состояниях:

  • индексированным(staget);
  • модифицированным(modifed);
  • зафиксированном(committed).

Моддифицированное состояние файла означает, что файл был изменен или добавлен в проект, но информация об изменениях еще не попала в базу данных Git. Индексированное состояние файла означает, что изменения в файле отмечены в облатси индексирования git и готовы к фиксации в базе данных репозитория


1.2. Состояние файлов в Git.

Прицип работы с Git таков:

  • Вы редактируете или создаете файлы в директории, которую отслеживает Git;
  • Git автоматически остслеживает все изменения, которые производились с файлами в данной директории, автоматически добавляя их в область индексирования;
  • Вы выполняете фиксацию изменений в проекте, перемещая изменения из области индексации в репозиторий.
